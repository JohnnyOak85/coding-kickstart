---
title: Version Control Glossary
---

### Branch

A parallel version of your repository. Branches allow you to work on different features or fixes without affecting the main codebase.

### Clone

A copy of a repository that you download from a remote server to your local machine. Cloning creates a complete replica, including all branches and history.

### Commit

A snapshot of changes in your repository. Commits are saved with a message describing what was changed.

### Conflict

Occurs when different branches modify the same part of a file, and Git is unable to automatically merge them. Conflicts need to be resolved manually.

### Fork

A copy of someone else's repository that you make under your own account. Forking is common when contributing to open-source projects.

### HEAD

A reference to the latest commit in the currently checked-out branch. It's essentially the "current position" in the branch.

### Merge

Combining changes from one branch into another. This is usually done after finishing work on a feature or fix.

### Merge Conflict

A situation that arises when Git cannot automatically reconcile differences between two branches. This requires manual intervention to resolve.

### Merge Request (MR)

GitLab's term for requesting to merge changes from one branch into another. Also known as a Pull Request (PR) in GitHub.

### Origin

The default name for the remote repository that a project was originally cloned from.

### Pull

Fetching changes from a remote repository and merging them into your current branch. It's like a combination of fetch and merge.

### Pull Request (PR)

GitHub's term for requesting to merge changes from one branch into another. Allows for code review and discussion before merging.

### Push

Uploading your local commits to a remote repository. Push sends your changes to the server where the repository is hosted.

### Rebase

A method of integrating changes from one branch to another by replaying your commits on top of another branch. Rebase creates a cleaner, linear history.

### Remote

A reference to a repository hosted on a server, such as GitHub or GitLab. You can pull from and push to remotes.

### Repository (Repo)

A storage space where your project’s code and history are kept. A repository can be local (on your machine) or remote (on a server).

### Staging Area

A place where you can group changes before committing them. Changes in the staging area are included in the next commit.

### Tag

A marker for a specific commit in your repository's history. Tags are often used to mark release points (e.g., v1.0).

### Tracking Branch

A local branch that is connected to a remote branch. This allows for easier synchronization between your local changes and the remote repository.

### Upstream

The main repository or original source of a forked repository. It’s where changes are typically pulled from.

### Working Directory

The files and directories that you see and work with in your project. The working directory reflects the most recent commit unless you have changes that are not yet committed.

### Git

A distributed version control system used to track changes in source code during software development. Git helps multiple developers collaborate on projects efficiently.

### GitHub

A web-based platform for version control using Git. GitHub offers additional features such as issue tracking, pull requests, and collaboration tools. It’s popular for open-source and private projects alike.

### GitLab

Similar to GitHub, GitLab is a web-based Git repository manager that provides tools for DevOps, including CI/CD (Continuous Integration/Continuous Deployment). GitLab can be hosted on your own servers or used via GitLab.com.

### CI/CD

Continuous Integration/Continuous Deployment. A practice where code changes are automatically tested and deployed to production after each commit or merge.

### Issue Tracker

A tool integrated with GitHub and GitLab that allows you to report and track bugs, feature requests, and other tasks related to a project.

### Kanban Board

A project management tool integrated into GitHub and GitLab that helps visualize and track tasks using columns and cards.

### Webhook

A feature in GitHub and GitLab that allows you to trigger external actions or notifications when certain events happen in the repository, such as pushing code or creating pull requests.
